感測器與致動器#

對應官方 Sensors 與 Actuators 章節。感測器從模擬中萃取觀測值（控制、RL、分析所需）；致動器把控制律計算的力矩寫回模擬。

感測器的共通模式#

  1. 初始化：用 model 建立感測器並指定量測對象。
  2. 更新：模擬迴圈中呼叫 sensor.update(state, ...)。
  3. 讀取：從感測器屬性取得結果（Warp 陣列）。
import warp as wp
import newton
from newton.sensors import SensorIMU

builder = newton.ModelBuilder()
builder.add_ground_plane()
body = builder.add_body(xform=wp.transform((0, 0, 1), wp.quat_identity()))
builder.add_shape_sphere(body, radius=0.1)
builder.add_site(body, label="imu_0")        # IMU 安裝點（site）
model = builder.finalize()

imu = SensorIMU(model, sites="imu_*")        # 1. 建立感測器

solver = newton.solvers.SolverMuJoCo(model)
state = model.state()

for _ in range(100):
    state.clear_forces()
    solver.step(state, state, None, None, dt=1.0 / 60.0)
    imu.update(state)                        # 2. 計算量測值
    acc = imu.accelerometer.numpy()          # 3. (n_sensors, 3) 線加速度
    gyro = imu.gyroscope.numpy()             #    (n_sensors, 3) 角速度

感測器會在建構時自動向 model 註冊所需的擴充屬性（如 body_qdd、Contacts.force），之後建立的 State／Contacts 就會包含它們——所以先建感測器、再配置 state。SensorContact 另需在 sensor.update() 前呼叫 solver.update_contacts()。

五種感測器#

感測器量測內容
SensorIMUsite 框架的線加速度與角速度
SensorContact剛體／形狀間的接觸力，含摩擦分解，可按對手方細分
SensorFrameTransform形狀／sites 相對於參考 sites 的相對變換
SensorTiledCamera跨多世界的光線追蹤彩色＋深度渲染（結果寫入傳入 update() 的輸出陣列）
SensorRaycast（已棄用）虛擬相機深度圖——請改用 SensorTiledCamera

Label Matching（標籤選取）#

多數感測器 API 用標籤模式選取 body／shape／joint／site，支援三種形式：

# 單一萬用字元模式（fnmatch：* 與 ?）
SensorIMU(model, sites="foot_*")

# 模式清單（聯集）
SensorContact(model, sensing_shapes=["*Plate*", "*Flap*"])

# 整數索引清單（直接選取）
SensorFrameTransform(model, shapes=[0, 3, 7], reference_sites=[1])

效能注意事項#

  • 感測器在 setup 階段建立一次、每步重用——讓 Newton 預先配置輸出陣列、避免每 frame 開銷。
  • 依賴擴充屬性的感測器（body_qdd、Contacts.force）會讓求解器每步多算這些量，即使當步沒讀取感測器也一樣——成本要算進去。

致動器（Actuators）#

實驗性：actuator API 可能變動。

致動器讀取模擬狀態、計算力矩（effort），並累加（scatter-add）到控制陣列——呼叫者每 frame 需先把輸出陣列清零。每個 Actuator 實例是向量化的：一個物件操作一批 DOF 索引，可高效整合到多平行環境的 RL 工作流程。三個可組合的構件（依序作用）：

構件作用
Delay（可選）把指令輸入延遲 N 個致動器時步，模擬通訊／處理延遲
Controller控制律，由狀態與目標算出原始力矩——PD、PID 或神經網路控制
Clamping[]（可疊加）馬達極限模型：ClampingMaxEffort（±對稱上限）、ClampingDCMotor（速度相關飽和）、ClampingPositionBased（位置相關查表）

每步管線：Delay 讀取 → Controller → Clamping → Scatter-add → 狀態更新。

# 建模時註冊，finalize 時自動實例化
builder.add_actuator(
    ControllerPD,
    index=dof_index,
    kp=100.0,
    kd=10.0,
    delay_steps=5,
    clamping=[(ClampingMaxEffort, {"max_effort": 50.0})],
)
model = builder.finalize()

Controller 與 Clamping 都可插拔——繼承基底類即可加入自訂模型；目標是在底層控制器支援時提供可微分與CUDA Graph 可捕捉的標準致動器模型。目前限制：尚無傳動模型（齒輪比）、僅支援 SISO（一 DOF 一致動器）、不含致動器動力學（慣性、摩擦、熱效應）。

搭配 Sites 使用#

Site（無碰撞參考框架）是感測器的安裝點與參考框架：IMU 裝在 site 上、FrameTransform 以 site 為參考。用 builder.add_site(body, label=...) 建立，或由 MJCF／USD 匯入。詳見核心概念・Sites。
