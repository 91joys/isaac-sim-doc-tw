偵測管線與五種策略#

對應 DeepWiki 2.1 Core Detection Engine 與 2.2 Protocol Definition and Content Matching。主要進入點是 ndpi_detection_process_packet()：每個封包依「先快後深」的順序，經過層層遞進的偵測策略。

先快後深的五層策略#

1. LRU 快取查詢 — O(1)7 種專屬快取（Ookla、BitTorrent、STUN、TLS 憑證、Mining、MS Teams、FPC DNS）—— 命中直接返回
↓ 未命中
2. Port 比對 — O(log n)查 tcpRoot / udpRoot 二元樹，只給出低信心的「猜測」（guessed_protocol_id）
↓
3. IP 比對 — O(k)查 Patricia 樹比對 CIDR 範圍（如 AWS、Cloudflare、TeamViewer 的已知網段）
↓
4. 字串比對 — O(n+z)Aho-Corasick 自動機比對 SNI / Host 等主機名稱樣式（如 *.netflix.com）
↓ 仍無法確認
5. 協定解析器（DPI）逐一呼叫候選 dissector 深度解析；失敗的加入 excluded 位圖不再重試
↓
結果確定ndpi_int_change_protocol() 寫入協定堆疊、設定信心等級、更新風險位圖

單一封包的處理流程#

  1. 解析封包標頭：ndpi_detection_parse_packet() 拆出 L3（IPv4/IPv6）與 L4（TCP/UDP/ICMP）；支援 GTP、CAPWAP、TZSP、VXLAN、GRE 等隧道協定的解封裝——偵測到隧道時遞迴處理內層封包。
  2. 檢查 flow 狀態：若 detection_completed 已設，跳到延伸處理。
  3. 執行偵測：依上方五層策略進行（check_ndpi_flow_func() 挑選並呼叫解析器）。
  4. 寫入結果：ndpi_int_change_protocol() 設定 detected_protocol_stack[] 與信心等級，必要時設 detection_completed。
  5. 延伸處理：若 extra_packets_func 有設，持續萃取中繼資料（TLS 憑證、HTTP 標頭、DNS 回應…）。

解析器的挑選與執行#

  • 註冊：初始化時 ndpi_callback_init() 把所有解析器塞進 callback_buffer[]；每個項目帶函式指標與 selection bitmask（TCP／UDP／IPv4v6）。
  • 啟用：ndpi_enabled_callbacks_init() 依 detection_bitmask 建立啟用清單；自訂協定一律啟用。
  • 執行：走訪啟用的解析器，跳過 flow 的 excluded_dissectors_bitmask 內已失敗者；解析器成功時呼叫 ndpi_set_detected_protocol()。
  • 排除：解析器判定「絕對不是我」時呼叫 NDPI_EXCLUDE_PROTO()，之後的封包不再重試該解析器。

放棄偵測（give up）#

處理了合理數量的封包仍無法辨識時，應用程式應呼叫：

ndpi_protocol result = ndpi_detection_giveup(ndpi_struct, flow);

它會做最後的啟發式判斷（port 猜測、IP 猜測），真的沒轍才回傳 NDPI_PROTOCOL_UNKNOWN，並可能標記 NDPI_UNDETECTED_PROTOCOL 風險。想知道結果是否來自猜測，可讀 flow->protocol_was_guessed。

效能設計重點#

  • 早停：一旦高信心辨識就不再往下跑。
  • 排除記錄：失敗的 dissector 記在 flow 位圖裡，之後直接跳過。
  • 快取重用：相似流量直接吃前次結果（confidence 標記為 DPI_CACHE）。
  • 選擇性啟用：只啟用設定需要的解析器。

自訂規則#

除了內建樣式，nDPI 支援用外部規則檔擴充偵測：可組合 port 範圍、payload 內容樣式與行為特徵來定義部署專屬的協定（ndpiReader -p 規則檔 或 ndpi_load_protocols_file()）。搭配 ndpi_add_host_ip_subprotocol()、ndpi_add_host_url_subprotocol() 也能在程式內動態加規則——見《擴充新協定》。
