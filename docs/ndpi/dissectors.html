協定解析器（Protocol Dissectors）#

對應 DeepWiki 3. Protocol Dissectors。解析器是針對單一協定的深度檢測模組，位於 src/lib/protocols/，共 80+ 個。每個解析器負責五件事：辨識協定、萃取中繼資料、子分類、標記風險、跨封包維護狀態。

解析器的組成#

元件說明例子
主搜尋函式主要偵測邏輯ndpi_search_stun()、ndpi_search_dns()

延伸解析函式後續封包的持續解析search_dns_again()、stun_search_again()
Keep Extra 函式判斷何時停止延伸解析keep_extra_dissection()

四種常見實作模式#

模式一：單包特徵驗證#

特徵明確、一兩個封包即可判定。例：RTP 檢查版本 == 2、payload type、長度欄位——全過就設協定並掛延伸解析，否則 NDPI_EXCLUDE_DISSECTOR。

模式二：多階段解析#

需要循序解析結構化資料。例：Kerberos 逐層解 ASN.1（sequence → pvno → msg-type → domain/hostname），任一步失敗即排除。

模式三：協定多工分流#

多協定共用同一連線時依首位元組分流。例：STUN 依 RFC 9443 的範圍：

首位元組判定
0–3STUN
16–19ZRTP
20–63DTLS（轉交 TLS 解析）
64–79TURN Channel Data（解出內層後遞迴）
128–191RTP
192–223RTCP

模式四：有狀態串流重組#

TCP 資料跨段時先緩衝再解析。例：Kerberos over TCP 遇到訊息長度大於單一封包時，配置緩衝區累積到完整再處理。

子分類：以 STUN 為例#

STUN 被大量 VoIP／視訊應用使用，解析器透過多種線索判斷「上面是誰」：

方法線索判定應用
屬性值0x0101、0x0103Zoom
屬性值0x4000–0x4007WhatsApp Call
屬性值0x8054、0x8095…MS Teams
屬性值0xFF03Google Call
Realm 欄位*.google.com / *.signal.org / *.facebook* / *.telegram*Google Call / Signal / Facebook / Telegram VoIP
Port10000Signal VoIP
LRU 快取先前判定過的端點沿用快取結果

最終堆疊為「master（STUN / DTLS / SRTP / RTP / RTCP）＋ app（WhatsApp / Teams / Signal…）」。

解析器的快取策略#

  • FPC DNS 快取（First Packet Classification）：DNS 回應「api.whatsapp.com → 192.0.2.1」時，把 IP → WhatsApp 存入快取；之後連往該 IP 的新連線第一個封包就能分類。
  • STUN 快取：key 為 (IP << 32) | port（IPv6 先雜湊）；儲存雙向端點、XOR-PEER 對端，並把 client port +1 / +2 一併入快取，涵蓋同一用戶端的相鄰連線。
  • BitTorrent 快取：記錄單端點 key 與雙端點合併 key 的 peer 關係，同一 client 換 port 也能認出。

最複雜的例子：QUIC 解密#

QUIC 的 Initial 封包雖然加密，但金鑰可從公開資訊（connection ID + 版本固定 salt）推導。nDPI 實際執行 RFC 9001 的推導流程來取出 SNI：

/* QUIC Initial 金鑰推導（RFC 9001）*/
initial_secret        = HKDF-Extract(initial_salt, connection_id)
client_initial_secret = HKDF-Expand-Label(initial_secret, "client in", "", 32)
key = HKDF-Expand-Label(client_initial_secret, "quic key", "", key_length)
iv  = HKDF-Expand-Label(client_initial_secret, "quic iv",  "", 12)
hp  = HKDF-Expand-Label(client_initial_secret, "quic hp",  "", key_length)

流程：移除標頭保護（AES-ECB / ChaCha20 算 mask）→ 推導金鑰 → AEAD（AES-GCM）解密 payload → 解析 CRYPTO frame → 重組 TLS ClientHello → 取出 SNI → 子分類應用。支援 Google QUIC（Q024+）、draft-23+、v1、v2 與 Facebook MVFST。

Monitoring 模式#

部分解析器在分類與 metadata 都完成後仍持續觀察流量做統計——STUN（偵測 RTP 媒體流與音視訊型態）、DNS（觀察後續查詢）、BitTorrent（hash 萃取）。狀態轉換：NORMAL → Extra Dissection → MONITORING。
