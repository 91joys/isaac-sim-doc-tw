MJWarp 效能調校與批次模擬#

整理自官方文檔 Performance Tuning、Batched Model Fields 與 Batch Rendering 章節。先讀 MJWarp 總覽與使用。

CUDA Graph capture#

mjw.step 這類 MJWarp 函式內部由一大串 kernel launch 組成，直接呼叫時 Warp 會逐一啟動。若函式會被重複呼叫（模擬迴圈幾乎必然如此），建議把整組操作捕捉成 CUDA graph：

with wp.ScopedCapture() as capture:
    mjw.step(m, d)

# 之後重複啟動 graph——通常比直接呼叫函式快非常多
wp.capture_launch(capture.graph)

細節見 Warp Graph API 參考。

批次上限：nconmax／naconmax／njmax#

接觸與約束上限在建立 mjw.Data 時（make_data／put_data）指定，記憶體與計算量都隨這些值線性成長——原則是「在不溢出的前提下越小越好」。好的值高度依賴環境，實務上靠試誤法迭代：

  • mjwarp-testspeed --measure_alloc：印出每步的實際接觸數與約束數。
  • mjwarp-viewer：互動操作場景，觀察是否出現 overflow 錯誤。

求解器迭代次數#

MuJoCo 預設的 iterations（求解迭代）與 ls_iterations（linesearch 迭代）上限通常已有合理效能。調低 Option.iterations／Option.ls_iterations 可能提升效能，但應該是調完接觸／約束上限之後的次要手段——調太低會讓約束求解器無法收斂，導致模擬不準或不穩定。

與 MJX 的差異：在 MJX（JAX）這些求解器參數是控制效能的關鍵；MJWarp 則在所有世界都收斂後可以提前退出（early exit）省掉多餘計算，所以這些設定對效能的影響相對小。

接觸感測器匹配上限#

含接觸感測器的場景有一個「每個感測器最多匹配幾個接觸」的參數 Option.contact_sensor_maxmatch——同樣是在不超限的前提下越小越好；超過上限的匹配接觸會被忽略。可以直接設定或用 XML 自訂數值欄位：

model.opt.contact_sensor_maxmatch = 16

<custom>
  <numeric name="contact_sensor_maxmatch" data="16"/>
</custom>

記憶體#

世界數很大時，模擬吞吐常被記憶體需求限制。最佳化考量：

  • CCD 碰撞器比基元（primitive）碰撞器耗記憶體（各配對用哪種碰撞器見 MuJoCo 的 pair-wise colliders 表）；multiccd 又比 CCD 更耗。
  • CCD 記憶體需求隨 Option.ccd_iterations 線性成長。
  • 含 mesh geom 且啟用 multiccd 的場景，記憶體隨「所有 mesh 中每面最大頂點數」與「每頂點最大邊數」線性成長。
  • mjwarp-testspeed --memory 回報總記憶體用量，以及 Model／Data 中吃記憶體的大戶欄位；CCD 與約束求解器的 inline 配置列為 Other memory。
  • 降低 CCD 上限：make_data／put_data 的 nccdmax／naccdmax 參數可設得比 nconmax／naconmax 小，只需涵蓋「CCD 碰撞器產生的接觸」上限。例：10 個世界共 80 個接觸——mesh-mesh 30（CCD）、橢球-橢球 10（CCD）、球-球 40（基元），則 nconmax／naconmax 至少設 8／80（broadphase 可能需要更多），nccdmax／naccdmax 設 3／30 即可。

每碰撞器接觸數差異：部分 MJWarp 碰撞器的最大接觸數與 MuJoCo 不同——PLANE<>MESH：4 vs 3；HFieldCCD：4 vs mjMAXCONPAIR。
稀疏性：稀疏 Jacobian 可望大幅省記憶體，進度見 #88。

大型場景：sleeping 與 islands#

總自由度（nv）很多的場景計算昂貴，但常常大部分物體是靜止的。MJWarp 可以把靜止物體睡眠（sleep），將其排除在多數計算的工作集之外；並把 body 分組成獨立的 island——island 內所有 body 都靜止時整個 island 一起入睡。目前碰撞管線與約束求解器都受益於 sleeping，未來會有更多元件支援。

Compact solver（緊湊求解器）#

針對「總 DoF 多、但活躍 DoF 少（通常 < 64，例如兩支帶夾爪的手臂 16 DoF ＋ 8 個活動物體 48 DoF）」的場景，MJWarp 提供利用睡眠機制的 compact solver：

  1. 由活躍 island 求出每個世界的活躍 DoF 集合。
  2. 把活躍 DoF 壓實（compact）進一塊已知最大尺寸（nvmax）的連續稠密工作區。
  3. 在壓實空間上用固定尺寸的 GPU tile 運算（如 blocked Cholesky 分解）執行 Newton 約束求解。
  4. 把結果散回（scatter）全域狀態，凍結非活躍 DoF。

固定尺寸的壓實工作區避免了 GPU 執行緒發散，並能吃到為固定 tile 尺寸最佳化的高效能 tensor／矩陣運算（參見 Tile 程式設計）。啟用步驟：

<!-- 1. 啟用 Newton 求解器；2. 啟用 sleep -->
<option solver="Newton">
  <flag sleep="enable"/>
</option>

# 或經 MjSpec
spec = mujoco.MjSpec()
spec.option.solver = mujoco.mjtSolver.mjSOL_NEWTON
spec.option.enableflags |= mujoco.mjtEnableBit.mjENBL_SLEEP

# 3. 配置 Data 時指定任一世界的最大預期活躍 DoF 數（nvmax）
d = mjw.make_data(mjm, nworld=2048, nvmax=64)

# 命令列
mjwarp-testspeed scene.xml --nvmax=64

nvmax 未指定時預設為完整 DoF 數（nv）；把它設成活躍 DoF 的緊上界能顯著降低 GPU 記憶體用量並提升吞吐。

提示：考慮把 sleep 容忍度從預設 0.0001 調高（XML sleep_tolerance="0.01" 或 spec.option.sleep_tolerance = 0.01），讓物體更快入睡。

批次 Model 欄位（Domain Randomization）#

為了讓不同世界用不同的模型參數，許多 mjw.Model 欄位帶有前導批次維度。預設維度為 1（field.shape[0] == 1），所有世界共用同一組值；可以用前導維度大於 1 的 wp.array 覆寫欄位，索引方式為模除：field[worldid % field.shape[0]]。

Graph capture 注意：欄位陣列（shape）必須在 wp.ScopedCapture 之前覆寫——重新指派不會反映到已捕捉的 graph。

# 作法一：capture 前直接覆寫 shape 與值
m.dof_damping = wp.array([[0.1], [0.2]], dtype=float)

with wp.ScopedCapture() as capture:
    mjw.step(m, d)

# 作法二：capture 前只覆寫 shape，capture 後用 wp.copy 更新「值」
m.dof_damping = wp.empty((2, 1), dtype=float)

with wp.ScopedCapture() as capture:
    mjw.step(m, d)

dof_damping = wp.array([[0.1], [0.2]], dtype=float)
wp.copy(m.dof_damping, dof_damping)   # 用 m.dof_damping = ... 重新指派則不會更新 graph

修改欄位#

修改 mjModel 欄位的建議流程是改對應的 mjSpec 再重新編譯出新 mjModel——但編譯需要 host 呼叫：每個新欄位實例 1 次，nworld 份就是 nworld 次。某些欄位可以不經編譯直接改、支援裝置上更新（哪些欄位安全見 mjModel changes；裝置上更新的擴充追蹤於 #893）。

Per-world meshes（每世界不同網格）#

Per-world meshes 讓不同世界模擬不同的 mesh，實現異質世界。工作流程：

  1. 建一個 mjSpec，內含全部 mesh 資產，以及所有變體中所需的最大 geom 槽位數。
  2. 逐一變異 spec 並 spec.compile()，編譯出每個變體。
  3. 編譯基準（base）模型，用它建立 mjw.Model。
  4. 用「從各變體收集來的每世界陣列」覆寫相關的 mjw.Model 欄位。

Geom 層級隨機化範例（1 個 body、1 個 geom、2 個 mesh 資產；世界 0-1 用 mesh_a、世界 2-3 用 mesh_b）：

nworld = 4

# 基準 spec：1 個 mesh geom＋全部 mesh 資產
spec = mujoco.MjSpec()
mesh_a = spec.add_mesh(); mesh_a.name = "mesh_a"
mesh_a.uservert = [0, 0, 0, 1, 0, 0, 0, 1, 0, 0, 0, 1]
mesh_b = spec.add_mesh(); mesh_b.name = "mesh_b"
mesh_b.uservert = [0, 0, 0, 2, 0, 0, 0, 2, 0, 0, 0, 2]

body = spec.worldbody.add_body(); body.pos = [0, 0, 1]
body.add_freejoint()
geom = body.add_geom(); geom.name = "obj"
geom.type = mujoco.mjtGeom.mjGEOM_MESH

# 編譯各變體，再還原並編譯基準模型
geom.meshname = "mesh_a"; mjm_a = spec.compile()
geom.meshname = "mesh_b"; mjm_b = spec.compile()
geom.meshname = "mesh_a"; mjm = spec.compile()

m = mjw.put_model(mjm)
d = mjw.make_data(mjm, nworld=nworld)

# 從各變體收集每世界的幾何／慣性欄位，覆寫到 mjw.Model
geom_id = mujoco.mj_name2id(mjm, mujoco.mjtObj.mjOBJ_GEOM, "obj")
variants = [mjm_a, mjm_b]
assignment = [0, 0, 1, 1]          # 每個世界用哪個變體

dataid = np.tile(mjm.geom_dataid, (nworld, 1))
geom_size = np.zeros((nworld, mjm.ngeom, 3))
# ...（geom_aabb、geom_rbound、geom_pos、body_mass、body_subtreemass、
#      body_inertia、body_invweight0、body_ipos、body_iquat 同理配置）

for w in range(nworld):
    ref = variants[assignment[w]]
    dataid[w, geom_id] = ref.geom_dataid[geom_id]
    geom_size[w] = ref.geom_size
    # ...（其餘欄位同理，從 ref 逐世界複製）

m.geom_dataid = wp.array(dataid, dtype=int)
m.geom_size = wp.array(geom_size, dtype=wp.vec3)
# ...（其餘欄位以下表的 dtype 覆寫）

Body 層級隨機化（不同世界的 body 有不同數量的 geom）作法相同，但基準 mjModel 必須帶著所有變體中最大數量的 geom 槽位；某變體用不到的槽位可以停用（contype=0、conaffinity=0、dataid=-1），但必須保留在基準模型的 body 裡。完整程式碼見官方文檔。

必須覆寫的批次欄位：

欄位dtypeShape
geom_dataidint(nworld, ngeom)
geom_sizewp.vec3(nworld, ngeom)
geom_aabbwp.vec3(nworld, ngeom, 2)
geom_rboundfloat(nworld, ngeom)
geom_poswp.vec3(nworld, ngeom)
body_massfloat(nworld, nbody)
body_subtreemassfloat(nworld, nbody)
body_inertiawp.vec3(nworld, nbody)
body_invweight0wp.vec2(nworld, nbody)
body_iposwp.vec3(nworld, nbody)

批次渲染（Batch Rendering）#

MJWarp 內建高吞吐批次光線追蹤渲染器，建構於 Warp 的 BVH 加速結構之上，可平行渲染多世界、多相機——在 NVIDIA GPU 上以每秒數百萬影格的速度渲染 MuJoCo 場景。主要能力：

  • Mesh 渲染＋貼圖：BVH 加速、完整材質貼圖支援；另有高度場（heightfield）與 flex 可變形體渲染。
  • 光照與陰影：動態光源、可設定陰影；可做 domain randomization：light_active、light_type、light_castshadow、light_xpos、light_xdir。
  • 異質多相機：每個世界多台相機，每台相機可有不同解析度（cam_resolution）、視野（cam_fovy、cam_sensorsize、cam_intrinsic）與輸出模式（cam_output）。
  • Domain randomization：可搭配批次 Model 欄位：geom_matid、geom_size、geom_rgba、mat_texid、mat_texrepeat、mat_rgba。
  • BVH 加速 ray casting：mjw.ray／mjw.rays 與 rangefinder 感測器。

基本用法#

渲染與 ray casting 需要 mjw.RenderContext——內含 BVH 結構、渲染專用欄位與輸出緩衝：

rc = mjw.create_render_context(
    mjm,
    nworld=1,
    cam_res=(256, 256),           # 覆寫相機解析度（或逐相機清單）
    render_rgb=True,              # 輸出 RGB（或逐相機清單）
    render_depth=True,            # 輸出深度（或逐相機清單）
    use_textures=True,            # 套用材質貼圖
    use_shadows=False,            # 陰影投射（較慢）
    enabled_geom_groups=[0, 1],   # 只渲染 group 0、1 的 geom
    cam_active=[True, False],     # 逐相機啟用／停用
    flex_render_smooth=True,      # flex 可變形體平滑著色
)

每個參數都可全域或逐相機套用，也能從 XML 解析（或經 mjSpec 設定）：

<camera name="front_camera" pos="3 0 2" xyaxes="0 1 0 -0.6 0 0.8"
        resolution="64 64" output="rgb depth"/>

渲染時先呼叫 mjw.refit_bvh 更新 BVH 樹，再呼叫 mjw.render 寫入輸出緩衝：

mjw.refit_bvh(m, d, rc)
mjw.render(m, d, rc)

輸出緩衝以 (nworld, npixel) 形狀堆疊所有相機的像素，RGB 打包成一個 uint32；RenderContext.rgb_adr／depth_adr 提供逐相機索引。便利函式 mjw.get_rgb／mjw.get_depth 直接回傳指定相機、全部世界批次的整形後資料：

nworld, cam_index = 1, 0
resolution = rc.cam_res.numpy()[cam_index]
rgb_data = wp.zeros((nworld, resolution[1], resolution[0]), dtype=wp.vec3)
mjw.get_rgb(rc, rgb_data=rgb_data, cam_id=cam_index)

基準與注意事項#

  • 渲染基準：mjwarp-testspeed benchmarks/primitives.xml --function=render；多場景結果見發布基準。
  • Mesh：渲染計算量隨 mesh 複雜度（頂點數、面數）成長——同尺寸下基元幾何比 mesh／高度場吞吐更高。
  • 縮放：渲染隨解析度（總像素數）與相機數線性擴展。
  • 完整範例見 tutorial notebook；渲染器的取捨與限制見差異頁。
