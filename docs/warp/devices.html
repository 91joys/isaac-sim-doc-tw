裝置・串流・並行#

整理自官方 Devices、Concurrency 與 Allocators 章節。

裝置模型#

Warp 用字串別名標識計算裝置：CPU 是 "cpu"，每張 CUDA GPU 是 "cuda:i"（i 為 CUDA 裝置序號）——與 PyTorch 慣例一致。每個 API 呼叫都可用 device 參數指定目標：

a = wp.zeros(n, device="cpu")
b = wp.zeros(n, device="cuda:0")
c = wp.zeros(n, device="cuda:1")
wp.launch(kernel, dim=b.size, inputs=[b], device="cuda:0")

"cuda:i" 對應裝置 i 的 primary CUDA context，與 CUDA Runtime API（PyTorch 等）相容，因此 GPU 記憶體可以直接共享。

預設裝置與 ScopedDevice#

省略 device 參數時使用預設裝置——初始化時有 CUDA 就是 "cuda:0"，否則 "cpu"：

wp.set_device("cuda:1")      # 改預設裝置（同時切換 CUDA context）
d = wp.get_device()          # 取得目前預設裝置
p = wp.get_preferred_device()  # 取得「原始」預設裝置

# 建議做法：用 ScopedDevice（可任意巢狀，離開時還原）
with wp.ScopedDevice("cuda:0"):
    b = wp.zeros(n)
    with wp.ScopedDevice("cuda:1"):
        c = wp.zeros(n)
        wp.launch(kernel, dim=c.size, inputs=[c])   # 在 cuda:1
    wp.launch(kernel, dim=b.size, inputs=[b])       # 在 cuda:0

特殊別名 "cuda" 指向目前的 CUDA context——方便讓外部程式（如 torch.cuda.set_device()）控制執行裝置;但在 context 可能被其他程式不可預期切換的環境，建議用明確的 "cuda:i"。進階者可用 wp.map_cuda_device("alias", ctx_ptr) 為自訂 CUDA context 註冊別名。

多 GPU 程式設計#

@wp.kernel
def inc(a: wp.array[float]):
    tid = wp.tid()
    a[tid] = a[tid] + 1.0

arrs = []
for device in wp.get_cuda_devices():      # 列出所有 CUDA 裝置
    with wp.ScopedDevice(device):
        a = wp.zeros(250 * 1024 * 1024, dtype=float)
        arrs.append(a)
        for _ in range(1000):
            wp.launch(inc, dim=a.size, inputs=[a])

wp.synchronize()                           # 等所有裝置完成

Peer access：硬體支援時（同型 GPU＋NVLink/PCIe 拓撲）可讓 GPU 直接讀另一張 GPU 的記憶體，初始化訊息會回報支援狀態。wp.is_peer_access_supported() 檢查、wp.set_peer_access_enabled() 啟用（有方向性）。好處是跨卡 DMA 直傳（免 CPU staging buffer）；代價是配置／釋放變慢——不需要就別開。注意 peer access 不會加速記憶體池配置器的傳輸，那要改開 memory pool access。

非同步執行#

CUDA kernel launch 對主機（Python 執行緒）非同步——wp.launch() 排程後立即返回，CPU 可同時做別的事;不同 CUDA 裝置上的 kernel 也能並行執行。CPU kernel launch 目前是同步的，要並行就先發 CUDA 再跑 CPU：

wp.launch(kernel1, ..., device="cuda:0")   # 非同步
wp.launch(kernel2, ..., device="cuda:1")   # 與 cuda:0 並行
wp.launch(kernel3, ..., device="cpu")      # CPU 同步執行，與兩張 GPU 並行
do_cpu_work()

CUDA 陣列的建立與初始化也非同步。陣列複製：主機↔裝置的複製只有在主機陣列是 pinned 時才非同步（wp.zeros(..., device="cpu", pinned=True)）——pinned 記憶體允許 DMA 直傳，適合作主機↔裝置的中轉緩衝，但配置較慢且系統有量限，因此 Warp 預設不 pin。

Stream 與 Event#

CUDA stream 是「在 GPU 上依序執行的操作序列」；不同 stream 的操作可以並行交錯。Warp 初始化時為每個 CUDA 裝置建立一條目前 stream，該裝置的所有操作都排進去。

stream = wp.Stream("cuda:0")             # 建立新 stream
s3 = wp.Stream("cuda:0", cuda_stream=h)  # 包裝外部 stream 句柄（與 PyTorch 共享）

# 用 ScopedStream 把一段操作排上指定 stream（兼具 ScopedDevice 功能）
with wp.ScopedStream(stream):
    a = wp.zeros(n, dtype=float)
    wp.launch(kernel, dim=n, inputs=[a])

典型用途——計算與傳輸重疊：

with wp.ScopedDevice("cuda:0"):
    compute_stream = wp.Stream()
    transfer_stream = wp.Stream()

    with wp.ScopedStream(compute_stream):
        wp.launch(kernel, dim=a.size, inputs=[a])   # 計算
    with wp.ScopedStream(transfer_stream):
        wp.copy(b, c)                               # 同時傳輸（c 需 pinned）

Event 提供裝置端的 stream 間同步（不阻塞 CPU）：在 stream A 上 wp.record_event(e)，讓 stream B wp.wait_event(e)——B 會等 A 到達該點才繼續，CPU 全程不停。stream.wait_stream(other) 是記錄＋等待的簡寫。Event 也能跨裝置使用，編排完全在 GPU 上運行的多 stream／多卡工作流。

同步指引#

原則：同步愈少愈好，但不能更少。過度同步扼殺並行；同步不足則產生亂序錯誤。主機端同步由重到輕：

函式等待範圍使用時機
wp.synchronize()所有裝置幾乎永遠不該用（除錯例外）
wp.synchronize_device(d)單一裝置的所有 stream等一張卡完成
wp.synchronize_stream(s)單一 stream等特定序列（如 readback stream）完成，其他照跑
wp.synchronize_event(e)單一 event 記錄點最細粒度——等到某個點即可，後續操作照常非同步

常見情況其實不需要手動同步——array.numpy() 等操作會自動同步（見 FAQ：何時該呼叫 wp.synchronize()）。Graph 擷取期間禁止主機端同步與同步版 default stream，只允許 event 同步;graph 擷取必須在同一條 stream 開始與結束，中間可以用 event fork／join 其他 stream（甚至其他 GPU），錄成多卡並行的單一 graph。

記憶體池配置器#

Warp 預設使用 CUDA 的 stream-ordered memory pool allocator（初始化訊息中的 mempool enabled）——配置／釋放依 stream 排序、可在 graph 擷取中配置記憶體，且從池中重配置遠快於原生 cudaMalloc。實務建議：

  • 迴圈內避免反覆配置：能預配置就預配置;或設定池的 release threshold（wp.set_mempool_release_threshold()）減少歸還開銷。
  • 跨裝置池存取：wp.set_mempool_access_enabled("cuda:0", "cuda:1", True) 讓一張卡直接存取另一張卡的池記憶體（等同池版 peer access）。
  • 自訂配置器：可掛接外部配置器,例如把 Warp 與 PyTorch 都導向 RMM,兩個框架共享同一個記憶體池，避免互相搶佔 GPU 記憶體。

CPU／GPU 跨裝置記憶體存取（integrated GPU、Grace Hopper 等統一記憶體架構）的能力檢查與存取模式選擇，見官方 CPU/GPU cross-device memory access。
