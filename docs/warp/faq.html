FAQ・除錯・限制#

整理自官方 FAQ、Debugging、Limitations 與 Profiling 章節。

常見問題#

Warp 與 Numba／Taichi／CuPy／PyTorch 是什麼關係？#

與 Numba／Taichi 同屬 kernel 級 GPU 程式設計，差異在於：Warp 提供 kernel 自動微分、以 C++/CUDA 為中間表示（易掛接 C++ 函式庫）、內建網格／稀疏體積／點雲／USD 等幾何資料結構。與 PyTorch／JAX 非競爭關係——張量運算用它們，稀疏／條件邏輯／異質負載用 Warp，透過 __cuda_array_interface__ 共享資料（詳見總覽文檔）。

該從 IsaacGym／PhysX 換成 Warp 嗎？#

不是替代品——Warp 的模擬能力主要服務需要可微分物理的開發者，不是完整物理引擎。Warp 也整合於 IsaacGym，適合做 RL 的獎勵／觀測計算等輔助工作。完整物理引擎見 Newton。

什麼時候該呼叫 wp.synchronize()？#

幾乎永遠不用。同步昂貴，Warp 會自動處理操作間的相依（如 array.numpy() 隱式同步）。唯一需要手動同步的情境：用 wp.copy() 非同步複製回 CPU 後、呼叫 .numpy() 之前。細節見同步指引。

為什麼 kernel 外不能對 Warp 陣列賦值？#

Python scope 的逐元素存取（array[i] = 1.0）需要極慢的裝置同步與複製,所以不支援。替代方案：從原生陣列初始化、launch kernel 設值,或用 wp.full(8, 1.23, dtype=float)、arr.fill_(1.23)、arr[:4].fill_(1.23)。

微分 wp.abs(x) 這類非平滑函式會怎樣？#

y=|x| 在 x=0 沒有唯一梯度（只有次梯度集合）;Warp 與多數自動微分框架一樣任選其一（wp.abs() 在原點取 1.0）。多數最佳化器（尤其隨機性的）對此不敏感。

支援多 GPU 嗎？#

支援——單一行程內即可跨多卡配置、launch 與複製（cuda:0／cuda:1 別名）;搭配 CUDA-aware MPI 的 mpi4py 也可以把 GPU 陣列直接傳給 MPI 呼叫。見多 GPU 程式設計。

如何回報問題？#

Bug／功能請求／技術問題用 GitHub Issues;其他事項寄 warp-python@nvidia.com。

除錯#

  • 環境診斷：python -m warp.utils --diagnostics 印出版本、驅動、裝置等資訊。
  • 印值：kernel 內 print(v) 支援全部內建型別;純量可用 C 風格 wp.printf("%f %d\n", x, i)。
  • 日誌：wp.config.log_level = wp.LOG_DEBUG、wp.config.verbose_warnings = True（含來源位置），也可掛自訂 logger。
  • Debug 模式：wp.config.mode = "debug"（或逐模組 wp.set_module_options({"mode": "debug"})）——啟用陣列越界檢查、tid 溢位警告、裝置碼行號。
  • 斷言：debug 模式下 kernel 內的 assert cond, "訊息" 生效，失敗時中斷並印出位置。
  • 單步除錯：CPU kernel 可用一般 C++ 除錯器（Visual Studio／gdb）;GPU 用 cuda-gdb。
  • 檢視生成碼:kernel 快取目錄裡有生成的 .cu／.cpp 原始碼,wp.config.line_directives = True 可映射回 Python 行號。
  • CUDA 錯誤驗證：wp.config.verify_cuda = True 在每次操作後檢查 CUDA 錯誤（慢，僅除錯用）。
  • 非有限值偵測：wp.config.verify_fp = True 檢查 NaN／Inf;搭配 wp.isnan()／wp.isfinite()。
  • CUDA 工具：compute-sanitizer（memcheck／racecheck）等 CUDA Toolkit 工具皆適用。

效能分析#

  • wp.ScopedTimer("label")：預設量 CPU 排程時間（不同步）;synchronize=True 量含裝置執行的總時間;use_nvtx=True 發 NVTX 範圍給 Nsight;cuda_filter 開啟 CUDA activity 分析（逐 kernel 計時）。
  • CUDA Event 計時：wp.Event(enable_timing=True)＋wp.get_event_elapsed_time() 精確量 GPU 區段。
  • Nsight Systems／Compute：系統級時間軸與單 kernel 深度分析;Warp 的 NVTX 標記讓時間軸更可讀。
  • 模組編譯分析：找出編譯耗時的 kernel，配合 AOT 工作流程預熱快取。

語言限制清單#

kernel 內不支援的 Python 特性#

lambda、list comprehension、例外處理、遞迴、eval()、動態資料結構（list／set／dict）——這些特性上 GPU 都是效能陷阱，Warp 刻意不支援以避免效能懸崖。

其他重要限制#

類別限制
kernel／函式字串不能傳入 kernel;wp.tid() 不能在使用者函式內呼叫;wp.atomic_add() 對 float16/bfloat16 需 compute capability 7.0+（否則回傳 0）;模組已載入後修改 wp.constant() 不會觸發重編譯
launch 網格每一維 ≤ 2³¹−1;超出 CUDA block 硬體限制時自動退回 grid-stride loop（可用 max_blocks 微調）
陣列最多 4 維;每維 ≤ 2³¹−1;無複數型別
struct成員不能是泛型（Any）;不支援繼承（用組合替代）
Volumetile 配置後拓撲不可變
多行程父行程的 CUDA context 不能用於 fork 的子行程（改用 spawn）;多行程共用 kernel 快取目錄可能出問題（每行程獨立目錄）
純量數學低精度型別的數學函式多以 float32 模擬;精度與平台相關細節見官方原文
變數作用域kernel 內變數作用域規則與 Python 略有差異（迴圈變數等）;見官方 Variable Scope
