MJWarp 與 MuJoCo 的差異・FAQ#

整理自官方文檔 Differences from MuJoCo 與 Frequently Asked Questions 章節。功能層級的相容性清單見總覽頁。

與 MuJoCo 的行為差異#

Warmstart#

未停用 warmstart 時，MJWarp 求解器一律用 qacc_warmstart 初始化加速度；MuJoCo 則會比較 qacc_smooth 與 qacc_warmstart 再決定用哪個。

慣性矩陣分解#

稠密計算時，MJWarp 的慣性矩陣分解 qLD 用 Warp 的 L'L Cholesky（wp.tile_cholesky）計算，與 MuJoCo 的反向 L'DL 例程（mj_factorM）結果不會逐位一致。

選項（Option）差異#

mjw.Option 欄位對應 mjOption，例外：

  • impratio 以其倒數平方根 impratio_invsqrt 儲存。
  • 求解器 tolerance 被鉗制在最小值 1e-6。
  • 接觸 override 參數 o_margin、o_solref、o_solimp、o_friction 不提供。

disableflags：mjDSBL_MIDPHASE 與 mjDSBL_AUTORESET 不提供；mjDSBL_NATIVECCD 的作用改為「把預設 box-box 碰撞器從 CCD 換成基元碰撞器」。
enableflags：mjENBL_OVERRIDE 與 mjENBL_FWDINV 不提供。

MJWarp 獨有選項：

選項用途
broadphasebroadphase 演算法類型（mjw.BroadphaseType）
broadphase_filterbroadphase 使用的過濾方式（mjw.BroadphaseFilter）
graph_conditional使用 CUDA graph conditional
run_collision_detection是否執行碰撞偵測例程
contact_sensor_maxmatch接觸感測器匹配的接觸數上限（見效能調校）

流體模型：修改 density、viscosity 或 wind 後可能需要更新 Model.has_fluid。
Graph capture：修改 mjw.Option 欄位後，可能需要重新 capture graph 設定才會生效。

SDF plugin#

SDF 碰撞支援 plugin：用 @wp.func 寫距離函式與其梯度，再覆寫模組層級的 hook。以 MuJoCo 的 bowl.xml 為例（節錄）：

import mujoco_warp as mjw
import warp as wp

@wp.func
def bowl(p: wp.vec3, attr: wp.vec3) -> float:
    """碗形的 signed distance function。attr = (height, radius, thickness)"""
    height, radius, thick = attr[0], attr[1], attr[2]
    width = wp.sqrt(radius * radius - height * height)
    q0 = wp.sqrt(p[0] * p[0] + p[1] * p[1])   # q = (norm_xy(p), p.z)
    q1 = p[2]
    if height * q0 < width * q1:
        dist = wp.sqrt((q0 - width) ** 2.0 + (q1 - height) ** 2.0)
    else:
        dist = wp.abs(wp.sqrt(q0 * q0 + q1 * q1) - radius)
    return dist - thick

@wp.func
def bowl_sdf_grad(p: wp.vec3, attr: wp.vec3) -> wp.vec3:
    """有限差分近似 SDF 梯度"""
    eps = float(1e-6)
    f0 = bowl(p, attr)
    return wp.vec3(
        (bowl(wp.vec3(p[0] + eps, p[1], p[2]), attr) - f0) / eps,
        (bowl(wp.vec3(p[0], p[1] + eps, p[2]), attr) - f0) / eps,
        (bowl(wp.vec3(p[0], p[1], p[2] + eps), attr) - f0) / eps,
    )

# 註冊：覆寫模組層級 hook
@wp.func
def user_sdf(p: wp.vec3, attr: wp.vec3, sdf_type: int) -> float:
    return bowl(p, attr)

@wp.func
def user_sdf_grad(p: wp.vec3, attr: wp.vec3, sdf_type: int) -> wp.vec3:
    return bowl_sdf_grad(p, attr)

mjw._src.collision_sdf.user_sdf = user_sdf
mjw._src.collision_sdf.user_sdf_grad = user_sdf_grad

物理 callback#

MuJoCo 用全域函式指標提供物理 callback；MJWarp 改為逐模型的 Python 函式，掛在 mjw.Model 實例的 Model.callback 上。可用的 callback：

Callback用途
control自訂控制律，寫入 Data.ctrl
passive自訂被動力，寫入 Data.qfrc_passive
act_dyn自訂致動器動態，寫入 Data.act_dot
act_gain自訂致動器增益，寫入 Data.actuator_force
act_bias自訂致動器偏置，寫入 Data.actuator_force
sensor自訂感測器，寫入 Data.sensordata；額外接收 stage 引數
contactfilter自訂接觸過濾，寫入 Data.contact
 
@wp.kernel
def _ctrl_callback(ctrl_out: wp.array2d(dtype=float)):
    worldid = wp.tid()
    ctrl_out[worldid, 0] = 2.0

def ctrl_callback(m, d):
    wp.launch(_ctrl_callback, dim=(d.nworld,), outputs=[d.ctrl])

m = mjw.put_model(mjm)
d = mjw.make_data(mjm)
m.callback.control = ctrl_callback
mjw.step(m, d)
assert d.ctrl.numpy()[0, 0] == 2.0

Box-box 碰撞#

box-box 預設走通用凸體碰撞管線（GJK／EPA）。設定 NATIVECCD 停用旗標可改用移植自 engine_collision_box.c 的專用基元碰撞器：

m.opt.disableflags |= mjw.DisableBit.NATIVECCD

專用碰撞器最多產生 8 個接觸點（凸體管線最多 4 個），對箱體堆疊與操作任務的接觸穩定性可能有幫助。

CCD margin#

某些 CCD 碰撞器不支援非零的 geom／pair margin，put_model 會拋 NotImplementedError：

Geom 配對情境解法
box-box、box-mesh、mesh-meshMULTICCD 啟用（預設開）margin 設 0 或停用 MULTICCD
box-boxNATIVECCD 啟用（預設開）margin 設 0 或停用 NATIVECCD

渲染器定位差異#

MJWarp 的批次渲染器與 MuJoCo 渲染器目的不同——它是高吞吐、低擬真的 single-hit raycaster。支援：簡單 Lambertian 漫反射著色、基本點光源／方向光、貼圖、陰影；不支援：全域光照等進階光效、物理式材質（PBR）。

FAQ#

學習框架#

MJWarp 能配 JAX 用嗎？可以，經 Warp 的 JAX 互操作；另外 MJX 以 impl='warp' 提供 MJWarp 部分 API 的 JAX 介面。

MJWarp 能配 PyTorch 用嗎？可以，經 Warp 的 PyTorch 互操作。

怎麼用 MJWarp 物理訓練策略？範例：Isaac Lab（經 Newton API）、mjlab（PyTorch 直接訓練）、MuJoCo Playground（經 MJX API）。

功能#

MJWarp 可微分嗎？不行。目前無法經 Warp 的自動微分取得梯度，進度見 #500。

支援多 GPU 嗎？支援，用 Warp 的 wp.ScopedDevice：

# 每張 GPU 各建一個 graph
graph = {}
for device in wp.get_cuda_devices():
    with wp.ScopedDevice(device):
        m = mjw.put_model(mjm)
        d = mjw.make_data(mjm)
        with wp.ScopedCapture(device) as capture:
            mjw.step(m, d)
        graph[device] = capture.graph

# 在每張 GPU 上啟動 graph
for device in wp.get_cuda_devices():
    wp.capture_launch(graph[device])

RL 分散式訓練實例見 mjlab distributed training。

GPU 上結果有決定性嗎？沒有。GPU 的非決定性 atomic 運算會造成順序或微小數值差異；要決定性結果請 wp.set_device("cpu") 改跑 CPU。GPU 決定性的開發追蹤於 #562。

方向（orientation）怎麼表示？單位四元數，遵循 MuJoCo 慣例 w, x, y, z（scalar, vector）。注意：MJWarp 用 Warp 內建的 wp.quaternion 型別，但不採用 Warp 的 x, y, z, w 慣例與運算，而是自行實作遵循 MuJoCo 慣例的四元數例程（見 math.py）。

有 named access／bind API 嗎？還沒有，追蹤於 #884。

沒有碰撞為什麼也回報接觸？凡是對任何碰撞感測器有貢獻的唯一 geom 配對，都會回報 1 個接觸——即使該配對並未真的碰撞。MuJoCo／MJX 的碰撞感測器是在計算感測資料時另外呼叫碰撞例程；MJWarp 則在主碰撞管線裡一併計算並把感測器資料存進 contacts。接觸感測器對「實際影響物理的接觸」回報的資訊是正確的。

Jacobian 為什麼一定是稠密的？稀疏 Jacobian 尚未實作，Data 的 ten_J、actuator_moment、flexedge_J、efc.J 一律是稠密矩陣（#88 追蹤中）。

陣列 shape 為什麼和 mjModel／mjData 不同？批次模擬下，許多 mjw.Data 欄位帶 Data.nworld 前導維度；部分 mjw.Model 欄位帶大小為 1 的前導維度（表示可被批次參數覆寫做 domain randomization）。另外 Model.qM、Data.efc.J、Data.efc.D 等欄位有 padding 以加速 GPU 載入。

MJWarp 和 MuJoCo 數值結果為什麼不同？MJWarp 用 float32，MuJoCo 的 mjtNum 預設是 double；求解器設定（含迭代）、碰撞偵測與小摩擦值都可能對浮點表示差異敏感。遇到非預期結果（含 NaN）請開 GitHub issue。

qM 稀疏性為什麼與 MuJoCo／MJX 不一致？語義不同：MuJoCo 的慣性矩陣永遠稀疏，mjtJacobian 影響的是約束 Jacobian；MJWarp（與 MJX）的約束 Jacobian 永遠稠密，mjtJacobian 被轉用來控制慣性矩陣的稠密／稀疏表示。且 AUTO 的自動稀疏門檻為 GPU 最佳化設在 nv > 32（MuJoCo／MJX 是 nv >= 60）；DENSE／SPARSE 明確設定則與兩者一致。此行為未來可能改變。

怎麼修模擬執行期警告？模擬中記憶體需求超過既有配置時會出警告：

  • nconmax／njmax 超限 → 調大 make_data／put_data 對應引數。
  • Option.ccd_iterations 超限（凸體碰撞迭代不足）→ 調大該設定；注意必須改在交給 put_model 與 make_data／put_data 的那個 mjModel 上。
  • Option.contact_sensor_maxmatch 超限 → 調大這個 MJWarp 專屬設定，或重構感測器匹配條件（已知兩個 geom 時直接指定 geom1／geom2）。
  • height field collision overflow（高度場潛在接觸數超過 mjMAXCONPAIR，部分接觸被忽略）→ 降低高度場解析度，或縮小與高度場互動的 geom。

編譯#

怎麼縮短編譯時間？減少需要走通用凸體碰撞管線的碰撞器種類（清單見 collision_convex.py 的 _CONVEX_COLLISION_PAIRS）；管線編譯時間的改善追蹤於 #813。

升級 MJWarp 後物理行為不對？Warp 快取可能與新程式碼不相容，除錯時先清掉：刪除 ~/.cache/warp 目錄，或：

import warp as wp
wp.clear_kernel_cache()

能提前編譯（AOT）而不是執行期編譯嗎？可以，見 Warp 的 AOT 編譯工作流程。
