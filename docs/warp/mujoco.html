MuJoCo Warp（MJWarp）總覽與使用#

整理自 MuJoCo 官方文檔 MJWarp 章節、google-deepmind/mujoco_warp GitHub README（v3.10）與 DeepWiki 架構解析。官方入門教學見 tutorial notebook（Colab）。

什麼是 MuJoCo Warp#

MuJoCo Warp（MJWarp）是用 NVIDIA Warp 重新實作的 GPU 加速版 MuJoCo 物理模擬器，針對 NVIDIA 硬體與大規模平行模擬最佳化，為機器人學研究提供高吞吐、高精度的模擬。它由 Google DeepMind 與 NVIDIA 共同開發維護，是 Newton Physics 專案的一部分，以 Apache-2.0 授權開源。

MJWarp 的所有物理演算法都以 @wp.kernel 撰寫、資料用 wp.array 存放在 GPU 上——也就是說，本站 Warp 文檔介紹的 kernel 編譯模型、CUDA Graph、PyTorch／JAX 互操作等能力，都直接適用於 MJWarp。

在 MuJoCo 生態系中的定位#

MuJoCo 生態系提供多種批次模擬方案，MJWarp 是其中的 CUDA 原生路線：

方案技術特性
mujoco.rolloutCPU 多執行緒呼叫 mj_step核心快、執行緒多的硬體可達高吞吐；但若模擬在 CPU、學習在 GPU（如 RL），host↔device 傳輸可能成為瓶頸
mjx.step（MJX）JAX vmap／pmap，XLA 編譯可跑 CPU／GPU／TPU 的多執行緒、多裝置模擬
mujoco_warp.step（MJWarp）Warp 直接生成 CUDA kernelNVIDIA GPU 上的多執行緒、多裝置模擬；接觸密集場景的擴展性優於 MJX 的 JAX 實作

與 MJX（JAX）的比較#

MJX 透過 JAX／XLA 取得 GPU 加速與自動微分，但也繼承了 JAX 的「sharp bits」；MJWarp 改走 Warp 直生 CUDA kernel 的路線，避開這些限制：

面向MJX（JAX）MJWarp（Warp）
變長陣列受 XLA 靜態 shape 限制動態配置，原生支援
條件邏輯受 XLA 控制流程約束kernel 內可寫自然的 Python 風格分支
除錯tracing／編譯讓除錯複雜kernel 內可用標準 Python 除錯
記憶體模型不可變陣列（函數式）可變陣列（指令式）
整合生態JAX 生態（Flax、Optax…）直接 CUDA、可獨立使用

MJWarp 同時被整合進 MJX 與 Newton 兩個框架，作為避開 XLA 限制的替代物理後端（MJX 以 impl='warp' 指定）。

何時該用 MJWarp？#

  • 高吞吐（適合）：MJWarp 為吞吐量（單位時間總模擬步數）最佳化，適合強化學習等需要大量樣本的應用。
  • 低延遲（不適合）：MuJoCo 為延遲（單步耗時）最佳化——同一個模擬，MJWarp 單步預期比 MuJoCo 慢。即時應用（MPC 線上控制、互動式遙操作）請用 MuJoCo。
  • 複雜場景（部分適合）：geom 或自由度很多的場景，MJWarp 擴展性優於 MJX、但不如 MuJoCo；超過約 60 DoF 可能出現明顯效能衰退。改善進度追蹤於：稀疏 Jacobian #88、block Cholesky #320、constraint islands #886、sleeping islands #887（緩解手段見大型場景與 sleeping）。
  • 可微分模擬（不支援）：MJX 的動力學 API 可經 JAX 自動微分；MJWarp 目前不支援經 Warp 的自動微分，進度見 #500。需要可微分物理請考慮 MJX 或 Newton 可微分模擬。

安裝與快速體驗#

MJWarp 需要 NVIDIA GPU 才能發揮模擬速度，但也支援 CPU 執行以便開發與除錯。

從 PyPI 安裝：

pip install mujoco-warp

從原始碼安裝（使用 uv）：

git clone https://github.com/google-deepmind/mujoco_warp.git
cd mujoco_warp
uv sync --all-extras
uv run pytest -n 8        # 跑測試確認一切正常

30 秒體驗——在本機看跳舞的人形機器人：

git clone https://github.com/google-deepmind/mujoco_warp.git && cd mujoco_warp
python benchmarks/run.py -f unitree_g1_flat --view

或直接在瀏覽器開 Colab 教學（免本機安裝）。官方 benchmark 場景涵蓋剛體接觸、可變形體、布料、SDF 等：Unitree G1（平地／高度場）、MyoArm、ALOHA（雜物抓取／鍋具／SDF／布料）、多人形機器人等，皆可用 python benchmarks/run.py -f <名稱> --view 試跑；每夜效能基準公開發布。

基本用法#

安裝後以 import mujoco_warp as mjw 匯入，struct、函式與 enum 都掛在頂層 mjw 模組。

Struct：把 Model／Data 搬上 GPU#

執行 MJWarp 函式前，要先用 mjw.put_model 與 mjw.make_data（或 mjw.put_data）把 struct 複製到裝置上：mjModel 上裝置後得到 mjw.Model，mjData 上裝置後得到 mjw.Data。

mjm = mujoco.MjModel.from_xml_string("...")
mjd = mujoco.MjData(mjm)
m = mjw.put_model(mjm)
d = mjw.put_data(mjm, mjd)

MJWarp 版本的 struct 鏡射 MuJoCo 對應結構，但有兩個關鍵差異：

  1. mjw.Model 與 mjw.Data 內含的是已複製到裝置上的 Warp 陣列。
  2. 不支援的功能對應欄位不存在於 mjw.Model／mjw.Data；若 mjModel 用到不支援的功能，put_model 會直接拋出例外。

批次大小參數#

MJWarp 為平行模擬最佳化，批次規模由這幾個參數指定：

參數意義
nworld要平行模擬的世界數
nconmax每個世界的預期接觸數；所有世界的接觸上限為 nconmax × nworld
naconmaxnconmax 的替代寫法——所有世界合計的接觸上限；兩者都設時 nconmax 被忽略
njmax每個世界的約束（constraint）數上限

語義差異：單一世界的接觸數可以超過 nconmax，只要全部世界合計不超過 nworld × nconmax；但每個世界的約束數嚴格受限於 njmax。
XML 不解析：nconmax／njmax 不會從 XML 的 size/nconmax、size/njmax 讀取（該參數已棄用），必須直接傳給 mjw.make_data 或 mjw.put_data。

函式#

MuJoCo 函式以同名、但符合 PEP 8 命名的 MJWarp 函式對外提供（如 mj_step → mjw.step）。主模擬流程（main simulation）的大部分函式與部分前向模擬子元件都可從頂層 mjw 模組取用。

最小範例#

# 用 100 種不同初速丟一顆球
import mujoco
import mujoco_warp as mjw
import warp as wp

_MJCF = r"""
<mujoco>
  <worldbody>
    <body>
      <freejoint/>
      <geom size=".15" mass="1" type="sphere"/>
    </body>
  </worldbody>
</mujoco>
"""

mjm = mujoco.MjModel.from_xml_string(_MJCF)
m = mjw.put_model(mjm)
d = mjw.make_data(mjm, nworld=100)

# 初始化各世界的速度
wp.copy(d.qvel, wp.array([[float(i) / 100, 0, 0, 0, 0, 0] for i in range(100)], dtype=float))

# 模擬物理
mjw.step(m, d)

print(f'qpos:\n{d.qpos.numpy()}')

命令列工具#

# 效能基準測試
mjwarp-testspeed benchmark/humanoid/humanoid.xml

# 互動式模擬檢視器
mjwarp-viewer benchmark/humanoid/humanoid.xml

# 效能剖析：輸出完整 event trace
mjwarp-testspeed benchmarks/humanoid/humanoid.xml --event_trace

mjwarp-testspeed 有豐富的選項（--help 查看），調校時常用的 --measure_alloc、--memory、--nvmax 見效能調校。開發者若要為 MJWarp 寫 Warp kernel，官方提供 kernel_analyzer VS Code 外掛（PR 上也會自動執行相同檢查）。

整合 MJWarp：選一條路#

多數情況下 MJWarp 可以直接當作 MuJoCo 的 drop-in 替代品安裝使用；若要接學習框架：

偏好方案說明
JAXMJX（impl='warp'）經 MJX 的 JAX API 使用 MJWarp；機器人 ML 範例見 MuJoCo Playground
PyTorchIsaac Lab（經 Newton）強大、高度可擴充的多物理模擬，深度整合 NVIDIA 生態系
PyTorch（輕量）mjlab把 Isaac Lab 的 manager-based API 直接架在 MJWarp 上，依賴極少、可直接存取原生 MuJoCo 資料結構

功能相容性#

MJWarp 支援 MuJoCo 絕大多數主模擬功能，目前的例外（v3.10）：

  • 積分器：不支援 IMPLICITFAST 的 midpoint 積分功能。
  • 求解器：PGS 與 noslip 尚未支援。
  • 致動器／感測器：PLUGIN 型別尚未支援。
  • Flex（可變形體）：實驗性——尚未全部實作或最佳化。
  • 可微分性：尚不可用（#500 追蹤中）。

把用到不支援功能的 mjModel 交給 put_model 會拋出例外。最新的相容性清單以 GitHub README 為準。其餘行為層面的細節差異（warmstart、選項、碰撞器……）整理於 MJWarp 差異與 FAQ。

架構速覽#

依 DeepWiki 的解析，MJWarp 架構由三個核心原則貫穿：

1. Host／Device 分離#

使用者照常用 MuJoCo 工具鏈編譯 MJCF XML 得到 mujoco.MjModel／MjData，再經 io 模組的 put_model／put_data／make_data 搬上 GPU、以 get_data_into 取回結果。Model 是不可變的靜態組態、Data 是可變的時變狀態；模擬迴圈完全留在裝置上，把昂貴的 CPU↔GPU 傳輸降到最少。

2. 批次平行執行#

所有 Data 陣列的第一維都是 nworld：單一共用的 Model 配上 N 份獨立世界的狀態，kernel 以 worldid 作為第一個執行緒索引啟動（wp.launch(dim=(nworld, ...))），每個執行緒（區塊）處理一個獨立模擬——這是 RL、蒙地卡羅與參數掃描高效平行化的基礎。

3. 與 MuJoCo 參考實作對齊#

MJWarp 維持與 MuJoCo C 實作的演算法相容性，測試套件在多樣場景下對照 MuJoCo 驗證結果——在拿到 GPU 效能的同時保證物理正確。
